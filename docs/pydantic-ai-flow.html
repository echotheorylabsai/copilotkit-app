Internals · L6 Shared-State Todo Agent

How one request flows: Browser → CopilotRuntime → AG-UI Adapter → Pydantic AI

A line-by-line trace of the todo agent. Every method, every transformation, every event — followed from the moment you type a message to the moment the panel re-renders. Two data directions are colour-coded throughout: inbound (browser → agent) outbound (agent → browser).

Traced against pydantic_ai 1.102.0 in .venv/…/pydantic_ai/ — code snippets simplified (docstrings & guards elided) for clarity
SCENARIO

You have the todo panel open with one item. You type “add buy milk” in the chat. The browser already holds shared state { todos: [{title:"walk dog"}] }. We trace how that state reaches the agent, how manage_todos runs, and how the new list streams back so the panel re-renders. This is the round trip we debug below.

02What actually crosses the wire

RunAgentInput is the one typed object every AG-UI request becomes — a single run's worth of input (thread id, shared state, message history, frontend tools) in one validated model. The runtime POSTs a single JSON body to /todos; AGUIAdapter.build_run_input parses it via RunAgentInput.model_validate_json. Only these fields matter for our trace:

# HTTP POST body → RunAgentInput
{
  "threadId": "t_abc",        ← thread id
  "runId":    "r_001",        ← run id
  "state":     { "todos": [{"id":"1","title":"walk dog","completed":false}] },  ← shared state
  "messages":  [ { "role":"user", "content":"add buy milk" }, … ],     ← history
  "tools":     [ { "name":"openOrCloseTodos", … } ]              ← frontend tools
}
Which process does the work? The Node CopilotRuntime is a pure forwarder for the todo agent — it relays this JSON to /todos unchanged (its A2UI / MCP / generative-UI middlewares are scoped to other agents, not "todo"). Every field is then parsed and mapped inside the Python agent process by pydantic_ai's AGUIAdapter — none of this happens in the runtime .ts.
FieldWhat the Python AGUIAdapter does with it
threadIdBecomes the run's conversation_id (reachable in tools as ctx.conversation_id); correlates the run.
runIdEchoed back in the outbound RunStartedEvent / RunFinishedEvent so the browser can match response ↔ request. Not passed into the run itself.
stateHydrated into ctx.deps.state (§04).
messagesload_messages → Pydantic AI ModelMessages, then sanitize_messages.
toolsThe browser's useFrontendTool, wrapped as an _AGUIFrontendToolset (an ExternalToolset): the model can call it, but the browser executes it (§06).

03inbound  The dispatch call stack

The adapter (AGUIAdapter) is the translator: it turns one AG-UI HTTP request into a Pydantic AI run, then turns the run's events back into AG-UI events (§07). One FastAPI route handler kicks off a fixed chain of its methods — read top-to-bottom: your handler (1) first, then each library call it triggers. Steps 2–5 are methods on UIAdapter in pydantic_ai/ui/_adapter.py; step 6 is where it hands off to the core agent. §09 shows this same machinery as a class design.

1
endpoint(request) → await AGUIAdapter.dispatch_request(request, agent, deps) backend/main.py
Your route handler. deps = deps_factory() — the factory (AGENT_DEPS["/todos"], bound once at route registration) builds a fresh StateDeps(TodoState()) per request.
↓ control enters the pydantic_ai library
2
UIAdapter.dispatch_request(…) dispatch_request
Entry point into the library. Builds this request's adapter via from_request, then returns a streaming response of run_stream(…).
3
UIAdapter.from_request(request, agent, …) from_request
Reads await request.body(); build_run_input parses the JSON into RunAgentInput. Captures the Accept header for SSE encoding.
4
adapter.run_stream(…) → transform_stream(run_stream_native(…)) run_stream
Two halves: run_stream_native produces native Pydantic AI events; transform_stream converts them to AG-UI events (§07).
5
adapter.run_stream_native(…) run_stream_native
The heart of inbound — (a) sanitize_messages, (b) attach the frontend toolset, (c) hydrate state into deps (§04), (d) call the agent.
6
async with agent.run_stream_events(deps=…, message_history=…, toolsets=…) agent.run_stream_events
Hands control to the core Pydantic AI run loop. From here the model is called and tools execute.
Security step you'd miss: sanitize_messages runs before the agent sees history, with the default manage_system_prompt='server'. It:
  • strips any client-sent SystemPromptPart — a client could otherwise inject instructions;
  • lets the agent's own system_prompt be re-injected as a separate step in run_stream_native via the ReinjectSystemPrompt capability;
  • drops dangling unresolved tool calls at the tail of client history.

04inbound  State hydration — how the browser's todos reach ctx.deps.state

State hydration = turning the inert JSON state dict that arrived in the request back into the typed object the agent's tools read (TodoState, reached as ctx.deps.state). On the wire, state is a plain dict; in memory it has to be a validated model. This block does exactly that conversion — the single most important one for shared state. It lives inside run_stream_native and decides whether to overwrite deps.state with the frontend's payload.

# inside UIAdapter.run_stream_native
if isinstance(deps, StateHandler):              # structural check, not subclass
    raw_state = self.state or {}                # self.state = RunAgentInput.state
    if isinstance(deps.state, BaseModel):
        state = type(deps.state).model_validate(raw_state)  # TodoState.model_validate({"todos":[…]})
    else:
        state = raw_state
    deps.state = state                          # overwrites the empty TodoState()
elif self.state:                          # state sent but deps isn't a StateHandler
    warnings.warn("…state was ignored…", UserWarning)  # real msg names the deps type, suggests StateDeps[...]
WhatDetail (verified)
StateHandler checkA @runtime_checkable Protocol. Passes if deps is a dataclass with a state getter/setter — StateDeps satisfies it structurally, not by inheritance.
self.stateThe AGUIAdapter.state cached property returns RunAgentInput.state, or None if it's missing or an empty {}.
Validationtype(deps.state) is TodoState, so TodoState.model_validate(raw_state) coerces the raw dict into a typed model — invalid shapes raise here.
Fresh per requestA factory lambda: StateDeps(TodoState()) — each request gets its own empty state, overwritten above. Why a factory, not a shared instance? See §10.

Inside the tool, the resolved value is reached through the RunContext Pydantic AI constructs for every call:

# backend/agents/todos.py — get_todos (the read tool)
@todo_agent.tool
def get_todos(ctx: RunContext[StateDeps[TodoState]]) -> list[dict]:
    return [t.model_dump() for t in ctx.deps.state.todos]   # ← [{title:"walk dog"}]

ctx.deps is the StateDeps instance · ctx.deps.state is the hydrated TodoState · RunContext is a dataclass the runtime fills — you never build it. deps is its only generically-typed field.

05Inside the agent — the model loop & manage_todos

The model reads history + system prompt, decides to call manage_todos with the new list, and Pydantic AI invokes the Python function. The return type — ToolReturn — is what makes shared-state sync possible.

# backend/agents/todos.py — manage_todos (the write tool)
@todo_agent.tool
def manage_todos(ctx: RunContext[StateDeps[TodoState]], todos: list[Todo]) -> ToolReturn:
    for todo in todos:
        if not todo.id: todo.id = str(uuid.uuid4())
    ctx.deps.state.todos = todos                       # mutate hydrated state (this request only)
    snapshot = {"todos": [t.model_dump() for t in todos]}
    return ToolReturn(
        return_value="Successfully updated todos",       # ← the LLM reads THIS string
        metadata=StateSnapshotEvent(type=EventType.STATE_SNAPSHOT, snapshot=snapshot),  # ← LLM never sees this
    )
Why ToolReturn has two channels.
  • return_value → becomes the tool-result message the model reads on its next step (“Successfully updated todos”).
  • metadata → documented as ”accessible by the application but not sent to the LLM” — the perfect side-channel for a protocol event.
Putting the event in return_value would waste tokens and confuse the model.

Note: ToolReturn also has a third field, content (a separate UserPromptPart); “two channels” is the scoped view for this trace.

06outbound  ToolReturn → ToolReturnPart → AG-UI events

Two hops turn that ToolReturn into the stream. First the run loop splits it into a message part; then the event stream inspects that part and emits the snapshot.

Hop 1 — the run loop builds a ToolReturnPart

# pydantic_ai/_agent_graph.py — after the tool function returns
if isinstance(tool_result, ToolReturn):
    tool_return = tool_result
else:
    tool_return = ToolReturn(return_value=tool_result)   # bare returns get wrapped

return_part = ToolReturnPart(
    tool_name=call.tool_name,
    tool_call_id=call.tool_call_id,
    content=tool_return.return_value,     # "Successfully updated todos"  → goes to the model
    metadata=tool_return.metadata,       # StateSnapshotEvent            → carried, NOT to the model
)

return_value→content, metadata→metadata — each field maps to its own slot in the part (why two channels: see §05). (Snippet trimmed: the real branch also rejects nested ToolReturns and sets a tool_kind field.)

Hop 2 — the event stream emits the snapshot

_handle_tool_result inspects every finished tool call: it always emits the model-visible result, then — if the part carries an event in metadata — forwards that straight into the AG-UI stream.

# pydantic_ai/ui/ag_ui/_event_stream.py — _handle_tool_result(...)
output = _tool_return_content(result)        # = result.model_response_str() = "Successfully updated todos"
yield ToolCallResultEvent(tool_call_id=result.tool_call_id, content=output)   # ① model-visible result

if isinstance(result, ToolReturnPart):
    possible_event = result.metadata or result.content      # metadata wins → StateSnapshotEvent
    if isinstance(possible_event, BaseEvent):
        yield possible_event                              # ② StateSnapshotEvent → straight into the AG-UI stream
    elif isinstance(possible_event, str | bytes):    # guard: a str is Iterable — don't char-iterate it
        pass
    elif isinstance(possible_event, Iterable):           # (a list of events also works)
        for item in possible_event:
            if isinstance(item, BaseEvent): yield item
Frontend-tool sidebar (openOrCloseTodos). A browser useFrontendTool arrives in RunAgentInput.tools and becomes an _AGUIFrontendToolset — an ExternalToolset with no Python body. When the model calls it:
  1. the run ends in a DeferredToolRequests output;
  2. the runtime ships the call to the browser, where your handler runs (setOpen(open) — open or close);
  3. the result returns in a follow-up POST.
So “add buy milk” can span two round trips: one that opens the panel, one that calls manage_todos.

07outbound  The SSE event stream the browser receives

The agent run emits Pydantic AI's own native events; transform_stream rewrites each into the AG-UI events the browser's protocol understands, streamed over SSE (a single long-lived HTTP response the server pushes events into). It wraps the run: before_stream() first, one handle_event() per native event, after_stream() last — all encoded as text/event-stream. For the manage_todos turn, the browser sees this ordered sequence:

One native event often fans out to multiple AG-UI events. transform_stream is the boundary where the translation happens.

Back in the browser, CopilotKit applies the StateSnapshotEvent to agent.state.todos; useAgent in TodoPage.tsx re-renders the <TodoList> with “buy milk” now present. The exact ordering of tool-call vs. text events follows whatever the model emits — but the result → snapshot pairing is fixed by the code above.

Why two cast() calls in dispatch_request

The selected code that triggered this section. Both casts are type-checker annotations only — zero runtime effect.

# pydantic_ai/ui/_adapter.py — dispatch_request
# The DepsT and OutputDataT come from `agent`, not from `cls`; the cast is necessary to explain this to pyright
adapter = cast(
    UIAdapter[RunInputT, MessageT, EventT, DispatchDepsT, DispatchOutputDataT],  ← outer cast: fix return type
    await cls.from_request(
        request,
        agent=cast(AbstractAgent[AgentDepsT, OutputDataT], agent),  ← inner cast: preserve agent's concrete params
        manage_system_prompt=manage_system_prompt,
        allowed_file_url_schemes=allowed_file_url_schemes,
        **kwargs,
    ),
)
cast()ProblemSolution
Outer from_request is a classmethod — pyright infers its return type from cls, which carries class-level type params. The actual concrete types (DispatchDepsT, DispatchOutputDataT) come from the agent argument at the call site. Cast overrides pyright's inferred return type to the correct concrete params.
Inner agent is passed in as a generic AbstractAgent; pyright would widen its type params when passing to from_request. Cast re-states the concrete params so pyright threads them through correctly.
The 20% that matters: the real work is from_request (parse request → build adapter). The casts paper over a limitation in how pyright handles generic classmethods — a type-level lie told to the checker so the rest of the call chain stays well-typed. They have no runtime effect whatsoever.

10Concurrency & isolation — why two users never collide

The natural worry: if User A and User B hit the /todos agent at the same instant, does A's state leak into B's run? No — and the guarantee is structural, not luck. Trace the lifecycle of every object each request touches. (State hydration — the mechanism that produces each request's own TodoState — is explained in §04.)

Both lanes feed the same Agent object on the right, yet carry different deps. The Agent is shared because it's read-only during a run; deps is per-request because it's the mutable part. Those two facts are the whole guarantee.

ObjectLifecycleWhy it's safe
Agent todos.pySHAREDBuilt once at import; holds tools, system prompt, model config. Read-only during a run — no per-call mutable fields, so concurrent reads can't race.
deps factory __init__.pySHAREDA lambda — a constructor, not an instance. Holds no state itself, so sharing it is harmless.
adapter ui/_adapter.pyPER-REQUESTfrom_request() mints a new adapter wrapping this request's run_input. self.state is a @cached_property off it — scoped to one request.
deps main.pyPER-REQUESTdeps = deps_factory() runs inside the handler, so each request gets its own StateDeps(TodoState()).
deps.state ui/_adapter.pyPER-REQUESTmodel_validate() builds a brand-new, deep-constructed TodoState (nested Todos rebuilt too), then rebinds. Shares no mutable object with the old default or the raw dict.
RunContext.deps _run_context.pyPER-REQUESTThreads this request's deps into every tool as ctx.deps. A's manage_todos can never see B's deps.
The single line that holds it all up. Every guarantee above collapses if deps is a shared instance instead of a factory. Pydantic AI hands you that choice on purpose — dispatch_request(deps=…) takes a value and never builds it for you, because only the app knows the right scope.
# backend/agents/__init__.py — the linchpin
AGENT_DEPS["/todos"] = lambda: StateDeps(TodoState())   # ✓ SAFE — new instance per request

# If it were a singleton, deps.state = state would race across users:
AGENT_DEPS["/todos"] = StateDeps(TodoState())             # ✗ NEVER — one object shared by all requests
Scope of the guarantee. Isolation here is per-request, not per-conversation. The backend keeps nothing between turns — persistence relies on the browser round-tripping todos back in each RunAgentInput.state (§02). In the AG-UI model the client is the source of truth for state; the server hydrates it fresh, runs, streams a snapshot back, and forgets.
